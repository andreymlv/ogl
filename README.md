# OGL — Game Engine

A 2D/3D game engine built with modern C++20, OpenGL, Qt 6, and EnTT.
Follows KDE Frameworks coding conventions throughout.

Reference: <https://learnopengl.com>

Awesome List: <https://github.com/stevinz/awesome-game-engine-dev>

---

## Goals

- Learn OpenGL incrementally following learnopengl.com
- Build a reusable engine that supports multiple games (Pong, Flappy Bird, …)
- Swap windowing backends (GLFW → X11 / Win32) and graphics APIs (OpenGL → Vulkan) without touching game code
- 100% autotest coverage on all engine classes
- TDD: write tests before implementation

## Architecture

### Dependency injection — no coupling between layers

`Application` receives a `Window` and a `Renderer` via `init()`.
It knows nothing about GLFW, OpenGL, or any platform.

```
main.cpp  (game)
  ├── QCoreApplication app(argc, argv)   // caller owns Qt lifecycle
  ├── GlfwContext ctx;  ctx.init()
  ├── auto window   = make_unique<GlfwWindow>();   window->init(800, 600, "Pong")
  ├── auto renderer = make_unique<OpenGLRenderer>();
  └── PongGame game;
      game.init(move(window), move(renderer))
      game.run()
```

To swap GLFW → X11: implement `Window`, inject `X11Window`.
To swap OpenGL → Vulkan: implement `Renderer`, inject `VulkanRenderer`.

### Game loop

```
Application::run()
  └── QTimer(0ms) → tick() every idle cycle
        ├── Window::shouldClose()  → quit if true
        ├── Window::pollEvents()
        ├── onUpdate(dt)           ← override in game subclass
        ├── Renderer::beginFrame()
        ├── onRender()             ← override in game subclass
        ├── Renderer::endFrame()
        └── Window::swapBuffers()
```

### Layer stack (planned — Step 9)

Each game pushes `Layer` objects onto the `LayerStack`.
`Application::onUpdate` and `onRender` forward to all layers in order.

```
LayerStack
  ├── GameLayer   ← physics, entities, rules
  └── HUDLayer    ← score, UI overlay
```

```cpp
class Layer : public QObject {
public:
    virtual void onAttach() {}
    virtual void onDetach() {}
    virtual void onUpdate(float dt) {}
    virtual void onRender() {}
};
```

Use Qt signals between layers (e.g. `collision(entityA, entityB)`) to keep them decoupled.

### ECS with EnTT (planned — Step 12)

```
Scene
  └── entt::registry
        ├── entity: bird
        │     Transform, RigidBody2D, BoxCollider2D, Sprite
        └── entity: pipe
              Transform, BoxCollider2D, Sprite

Scene::update(dt, renderer)
  ├── PhysicsSystem::update(registry, dt)
  ├── CollisionSystem::update(registry)     → emits Qt signals
  └── RenderSystem::update(registry, renderer2D)
```

**Components** — plain data structs, no methods:

```cpp
struct Transform    { glm::vec2 position, scale; float rotation; };
struct RigidBody2D  { glm::vec2 velocity; float gravityScale; };
struct BoxCollider2D{ glm::vec2 size; bool isTrigger; };
struct Sprite       { shared_ptr<Texture> texture; glm::vec4 color; int zOrder; };
struct Tag          { string name; };
```

**Systems** — free functions, no state:

```cpp
namespace PhysicsSystem  { void update(entt::registry &, float dt); }
namespace CollisionSystem{ void update(entt::registry &); }
namespace RenderSystem   { void update(entt::registry &, Renderer2D &); }
```

### Renderer2D (planned — Step 7)

Batched quad/sprite renderer. All 2D games use this.

```cpp
class Renderer2D {
public:
    void beginScene(const Camera2D &);
    void drawQuad(glm::vec2 pos, glm::vec2 size, glm::vec4 color);
    void drawSprite(glm::vec2 pos, glm::vec2 size, const Texture &);
    void endScene();   // flushes draw call batch
};
```

## Coding Conventions

Follow [KDE Frameworks Coding Style](https://community.kde.org/Policies/Frameworks_Coding_Style).

### Naming

| Element | Convention | Example |
|---|---|---|
| Class / struct | `PascalCase` | `GlfwWindow` |
| Member variable | `m_` prefix + `camelCase` | `m_valid`, `m_frameTimer` |
| Static member | `s_` prefix + `camelCase` | `s_instance` |
| Method / function | `camelCase` | `shouldClose()`, `beginFrame()` |
| Namespace | `lowercase` | `engine`, `pong` |
| File | `lowercase.h` / `lowercase.cpp` | `glfwwindow.h` |

### Class rules

- No logic in constructors — only member initialization
- No exceptions — use `bool init()` + `bool isValid()` pattern
- No raw owning pointers — use `unique_ptr` / `shared_ptr`
- pImpl for all non-trivial classes: `Private` forward-declared at **namespace scope** (not nested), defined only in `.cpp`
- Non-QObject pImpl: use direct `d->` access (do NOT use `Q_DECLARE_PRIVATE` — it requires Qt headers in every consumer and the nested forward-decl pattern causes incomplete-type linker errors)
- QObject subclasses: add `Q_OBJECT`, use `Q_DECLARE_PRIVATE` + `Q_D()`
- Every public class in `libengine.so` must be marked `ENGINE_EXPORT`

```cpp
// header — only the contract, Private is incomplete here
class FooPrivate;          // forward at NAMESPACE scope, not nested
class Foo {
    std::unique_ptr<FooPrivate> d;
};

// .cpp — full definition, compiler sees complete type
class FooPrivate { ... };
Foo::Foo() : d(std::make_unique<FooPrivate>()) {}
void Foo::method() { d->m_field; }
```

### Headers

- Use `#pragma once`
- Forward declare instead of including wherever possible:
  - `T*`, `T&`, `unique_ptr<T>` parameters → forward declare `T`
  - Inheriting `T` or holding `T` by value → must include
- Do NOT list `.h` files in `target_sources` — ECM picks them up automatically from the source directory

### Include order (enforced by `.clang-format`)

```cpp
#include "own_header.h"      // 1. own header first

#include "engine/sibling.h"  // 2. same library

#include <QObject>           // 3. Qt headers
#include <QTimer>

#include <memory>            // 4. standard library
```

### Logging

Use `ecm_qt_declare_logging_category` in CMake. Each library gets one category.
Log with `qCDebug(Engine)`, `qCWarning(Engine)`, `qCCritical(Engine)`.
Never use `qDebug()` / `qWarning()` directly — they lack category filtering.

### Shared library

- Every public class needs `ENGINE_EXPORT` (from generated `engine_export.h`)
- Build always uses shared libs (`BUILD_SHARED_LIBS=ON` is ECM default via `KDECMakeSettings`)
- `-fvisibility=hidden` is set by KDE compiler settings — without `ENGINE_EXPORT` symbols are invisible to test binaries

### CMakeLists.txt structure

- One `CMakeLists.txt` per directory
- Parent file owns the target definition (`add_library`, `target_link_libraries`)
- Child `src/CMakeLists.txt` only calls `target_sources(engine PRIVATE ...cpp files...)`
- `autotests/CMakeLists.txt` uses `ecm_add_test` — no need for `add_test` manually
- `if(BUILD_TESTING)` guards the autotests subdirectory

## Build

### Prerequisites

- CMake 4+, Ninja, clang (with clazy)

### Configure

```sh
cmake --preset sanitizers   # ASan + UBSan + LSan

cmake --preset release
```

```sh
cmake --build build/sanitizers --target engine
cmake --build build/sanitizers --target applicationtest
```

```sh
cd build/sanitizers
ctest -R applicationtest -V
```

## Next Step: Hello Triangle (Step 6)

### What to build

A `GlObject<Deleter>` RAII template and a `Shader` class.
`OpenGLRenderer::beginFrame()` will use the shader to draw a triangle.

### `GlObject` RAII template

```cpp
// engine/src/globject.h
template<auto Deleter>
class GlObject
{
public:
    explicit GlObject(GLuint id) : m_id(id) {}
    ~GlObject() { if (m_id) Deleter(m_id); }

    GlObject(GlObject &&o) noexcept : m_id(std::exchange(o.m_id, 0)) {}
    GlObject &operator=(GlObject &&o) noexcept {
        if (this != &o) { if (m_id) Deleter(m_id); m_id = std::exchange(o.m_id, 0); }
        return *this;
    }
    GlObject(const GlObject &) = delete;
    GlObject &operator=(const GlObject &) = delete;

    GLuint id() const { return m_id; }
    explicit operator bool() const { return m_id != 0; }

private:
    GLuint m_id = 0;
};

// Concrete aliases
inline void deleteProgram(GLuint id) { glDeleteProgram(id); }
inline void deleteBuffer(GLuint id)  { glDeleteBuffers(1, &id); }
inline void deleteVao(GLuint id)     { glDeleteVertexArrays(1, &id); }

using ShaderProgram = GlObject<deleteProgram>;
using GlBuffer      = GlObject<deleteBuffer>;
using Vao           = GlObject<deleteVao>;
```
